A plugin framework with three seams

Beacon is one container, one process: a scheduler fires report pipelines and an htmx web viewer serves the results. Everything platform-specific, format-specific, or channel-specific lives behind a plugin seam: collectors gather, renderers shape, notifiers announce.

The three contracts

Collector
wkx_beacon.collectors
  • Declares its report type and platform, e.g. cost × aws.
  • Returns a typed ReportData model it owns; the data contract for that report type.
  • Contributes a template pack so the html renderer can shape its report type.
Renderer
wkx_beacon.renderers
  • Turns ReportData into one or more artifacts (an HTML page now).
  • html dispatches on report type to the contributed template pack.
  • Generic renderers (a future json) serialise any report type.
Notifier
wkx_beacon.notifiers
  • Receives a run summary: report name, headline figures, artifact link.
  • Delivers it on a channel; email-ses first.
  • Never sees raw platform data.

The MVP plugins one of each, exercising every seam

aws-cost
collector · cost × aws
  • One AWS Cost Explorer call per run: daily granularity, unblended cost, current month plus previous 30 days, grouped by the Service and Env cost-allocation tags (two group-bys per call; totals derived by summing).
  • Computes month-to-date, yesterday, per-tag breakdowns, untagged share, and projected month-end (average of complete days, extrapolated; no forecast API).
  • About USD 0.30/month in Cost Explorer request charges at one call per day.
  • Returns CostReportData; config: budget, currencies, group-by tag keys.
html
renderer
  • Renders the collector's template pack into a fully self-contained artifact: inline CSS, inline SVG/HTML charts, CSS-only tooltips.
  • No JavaScript, no external assets; identical from the store, the viewer, or a saved file.
  • Renders no account identifiers; the page is public.
email-ses
notifier · Amazon SES v2
  • Success: subject carries the headline ($4.65 MTD, projected $48.05 of $50.00), body is the summary plus the stable /latest link.
  • Failure: subject names the failed stage.
  • Sender beacon@wingkongexchange.dev; the SES domain identity and its Cloudflare DNS records are a platform prerequisite (Terraform in wkx-platform), alongside beacon's IAM permissions.
Source of truth is fixed; display is per-report config. Two knobs, same pattern.
Currency: AWS Cost Explorer only returns USD. display = "usd" (default) reports purely in USD; display = "local-first" shows local-currency figures first with USD alongside, converted at a configured static rate shown in the report fine print. This deployment: NZD first at 1.64 NZD/USD.
Days: billing data is always UTC days and UTC months (the invoice's frame; projections and budget maths use complete UTC days). day_display = "utc" (default) labels dates as UTC; day_display = "local-first" labels each UTC billing day with the local date covering most of it (UTC 3 Jul → "4 Jul" in NZ), with both dates in tooltips and the table, and the mapping stated in the fine print. The morning-of-the-1st report wraps up the previous billing month by design. This deployment: local-first, Pacific/Auckland.

Runtime scheduler, failure model, store, health

Scheduler with opt-in boot catch-up

APScheduler in the FastAPI process, one timezone-aware cron trigger per report, max_instances=1 so a slow run never stacks on itself. Boot catch-up is per-report config (catch_up = true), default off. When enabled: on boot, a report whose last successful run predates its previous scheduled fire time runs immediately. When off, a missed fire stays missed until the next cron.

07:00 · DUE
Host down. Cron fires into the void.
09:12 · BOOT
Beacon starts. Last success < previous fire time.
09:12 · CATCH-UP
Only if catch_up = true. Report lands late, not never.
07:00 +1d
Normal cadence resumes.

A failed run is still a run

Stages run in order; third-party exceptions are translated at the boundary into a BeaconError hierarchy. Within a run, boto3's standard retry mode absorbs transient blips; there is no retry queue, the next cron is tomorrow.

collectrenderstorenotify
any stage raises · CollectError / RenderError / NotifyError
Recorded, visible, announced.
The run lands in the store with status and error summary, shows in the viewer, and notifiers send a failure notice instead of a report. You hear about breakage the same way you hear about spend. A notify failure is logged and recorded on the run.

Store: filesystem, no database

run.json is written last as the commit marker; the viewer ignores run directories without one. An in-memory index scans at startup and updates after each run. Retention is keep-everything for now.

{data_dir}/reports/<report-name>/runs/<run-id>/
    run.json       # status, stage outcomes, timings,
                   # headline summary · written last
    artifacts/
        report.html

Health and observability

Run records are the metrics. No metrics endpoint in the MVP.

GET /healthz Docker HEALTHCHECK JSON logs → stdout report + run_id fields /wkx/beacon/<env>

/healthz checks the scheduler is running and the data dir is writable. Logs carry report and run_id at every stage boundary and flow to CloudWatch via the platform agent.

The htmx viewer read-only over the store; it never triggers runs

Server-rendered Jinja2 + htmx fragments. htmx is vendored into the image, no CDN at runtime, no JS build chain.

Artifacts are self-contained HTML (inline CSS, no external assets), rendered at generation time: identical from the store, in the viewer, or saved to disk years later.

Public-page hygiene: templates render no account IDs or ARNs, only tag values and dollar figures. Conservative security headers (CSP, no-referrer, nosniff). Authn stays a platform decision.

Wired by config beacon.toml declares reports; plugins validate their own blocks

[[report]]
name       = "platform-cost"
collector  = "aws-cost"
renderers  = ["html"]
notifiers  = ["email-ses"]
schedule   = "0 7 * * *"   # cron, in the configured timezone
timezone   = "Pacific/Auckland"

[report.collector_config]      # validated by the aws-cost plugin's model
budget          = 50.0
budget_currency = "NZD"
display         = "local-first"  # currency display, default "usd"
fx_usd_to_local = 1.64           # static, shown in report fine print
day_display     = "local-first"  # date labels, default "utc";
                                   # data is always UTC billing days

[report.notifier_config.email-ses]
to = ["you@example.com"]
Fails at boot, not at 07:00. Unknown plugin names, unknown config keys (extra="forbid") and missing entry points are startup errors that list what was found versus what was expected.

Typer, per the conventions. Logging configured once at the entry point; JSON in the container, dev format locally.

Testing

No live AWS anywhere. Cost Explorer and SES run against botocore's Stubber. The renderer gets golden-file tests from fixture CostReportData. Pipeline failure paths use deliberately-failing fake plugins registered through real entry points, which also proves the third-party discovery path. Viewer routes and htmx fragments via FastAPI's TestClient over a fixture store.

Plugin conformance checks ship as a reusable test kit: the plugin API is the product, so third parties get its tests.

CI and platform prerequisites

CI: ruff check + format check, ty, pytest with coverage, arm64 Docker build. The deploy workflow (GitHub OIDC → ECR → SSM send-command) is inherited from the platform reference project when the platform is ready; until then docker compose up locally, which is also beacon's Docker Compose platform story.

Terraform work in wkx-platform: SES domain identity + Cloudflare DNS records; IAM permissions for ce:GetCostAndUsage and ses:SendEmail (mechanism per the deferred platform decision); SSM params under /wkx/beacon/<env>/; data dir; log group.
